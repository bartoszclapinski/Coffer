- `--:--` sprint planned — next-month expense forecast per category, feeding a suggested budget limit. The "forecasting next month's expenses" beyond-roadmap item. Complements Sprint 20/21 (which track/alert on the *current* month) by *anticipating* the next one. Sits entirely on existing machinery: Sprint-16 `RecurringFlow` (`MatchCategoryId` + cadence) for the fixed side, Sprint-18 `IVariableBurnQuery`'s recurring-flow exclusion for the variable side, Sprint-19/11 per-category / per-month aggregations for history, and Sprint-20 `SetBudgetAsync` for the one-click "accept suggestion". No migration, no AI, no new cost.
- `--:--` decision: **forecast = fixed + variable, per category, for `anchor + 1 month`** (anchor = latest transaction's month, dashboard convention). Fixed = active outflow flows attributed by `MatchCategoryId` that land in the target month (monthly always; interval > 1 by cadence phase), summed by `TypicalAmount`; null category → an unattributed-fixed line. Variable = trailing-3-month per-category debit magnitude, **excluding transactions whose merchant matches an active flow** (finer than Sprint-18's whole-category exclusion, so a category holding one recurring merchant keeps its other discretionary spend), averaged `/ 3`. Suggested limit = total rounded **up to the nearest 10 zł** for headroom. Explicitly a *simple* projection (flat averages + known recurring charges) — no seasonality/trend, labelled as such; accrual offset ignored (forecast by payment month, matching how budgets count spend).
- `--:--` decision: **pure engine in a new `Coffer.Core/Forecasting`** (`ExpenseForecastEngine` combines assembled per-category inputs + rounds the suggested limit); the Infrastructure `ExpenseForecastQuery` only assembles inputs (flows → per-category fixed via an `OccursInMonth` cadence helper; trailing per-category variable with merchant exclusion; active budgets for the current limit). Mirrors the Sprint-20 engine/query split. New `AddCofferForecasting` DI method; tests over `PlanningDbTestBase`.
