# Sprint 24 — Restore from a local snapshot

**Phase:** — (roadmap-adjacent; `docs/architecture/08-backup-and-recovery.md`, the "Restore flow" branch for a local daily snapshot. Restore-from-BIP39-seed and the DEK dual-wrap format it needs are explicitly out of scope — see Deferred; they are Sprint 25.)
**Status:** Planned
**Depends on:** sprint-23 (`IBackupService` / `BackupService` / `BackupSnapshotWriter` / `BackupRetention` — the snapshot files and their naming), sprint-4 (`MigrationRunner` + `IPreMigrationBackup` + startup migration orchestration in `App.axaml.cs`), sprint-2/3 (`IVaultPaths`, `coffer.db`), sprint-15 (i18n). No new schema.

## Goal

The owner can recover from a bad day — a corrupt write, an accidental mass re-categorisation, a botched import — by restoring the encrypted database from a chosen **daily snapshot** through a guided flow, instead of hand-copying a `.db` file. Concretely: a **"Restore from snapshot…"** action in the Settings "Backup & Recovery" panel opens a picker listing the available dated snapshots (newest first); the owner picks one and confirms; the app takes a **safety snapshot of the current database first**, then swaps the chosen snapshot into place and restarts into it. Same vault, same master password, same DEK — no key or password interaction at all. Deterministic; no AI, no schema change, no network.

## Why this sprint exists

Sprint 23 built the safety net (daily snapshots + rolling retention + archive export) but left the other half open: there is **no way to actually use a snapshot from inside the app**. The Settings panel still says *"Guided restore is coming. For now, a snapshot can be restored by copying it back over the database file while logged out."* — the owner has to close the app, find `%LocalAppData%\Coffer\backups\`, pick the right dated file by hand, delete the live `coffer.db` + its `-wal`/`-shm` side-files, copy the snapshot's set into place, and hope they got the WAL set consistent. That is exactly the kind of manual, error-prone step around irreplaceable financial data this app exists to remove. A backup you can't confidently restore is only half a backup. This sprint makes the local-snapshot restore a guided, reversible, one-confirmation action — the common recovery case — and defers only the rarer forgot-my-password seed-recovery case (which needs a crypto-format change) to Sprint 25.

## Design decisions (the shape we commit to)

- **Local snapshot restore only; seed restore is Sprint 25.** Restoring from a daily snapshot is *the same vault* — the snapshot's `coffer.db` is encrypted with the same DEK, unlocked by the same master password. So restore here is a pure **file swap of already-encrypted bytes**: no DEK re-derivation, no password re-entry, no `dek.encrypted` change, no crypto at all. Restore-from-BIP39-seed is a different animal — it needs a `dek.encrypted` format that carries a *seed-wrapped* copy of the DEK (v1 stores only a password-wrapped copy; `SetupService` receives the mnemonic but never uses it), plus an "enable seed recovery" upgrade for the existing vault and a new-password re-encrypt step. That is a self-contained crypto sprint (25); keeping it out here keeps this sprint free of any key handling.
- **Deferred restore, applied at next startup while the DB is closed.** You cannot safely overwrite `coffer.db` while it is open — SQLite holds a file lock and a live WAL, and the app opens the DB lazily on first `CofferDbContext` use. So restore is **two-phase**: from Settings (logged in) the owner only *stages* a restore — the app validates the chosen snapshot and writes a small **restore-request marker** (`restore-pending.json` in `%LocalAppData%\Coffer\`), then tells them a restart is required. On the **next startup**, `App` checks the marker **before opening the DB or logging in**, performs the swap against a closed file, deletes the marker, and continues to the normal login. This is the Windows "pending file rename" pattern; it sidesteps the file-lock/WAL-consistency problem entirely and is deterministic and testable (no live DB handle in play). It also mirrors where the app already does pre-login startup work (`ResolveStartupWindow` / `RunStartupMigrationAsync`).
- **The swap is itself reversible — safety snapshot first.** Before overwriting the live `coffer.db`, `ApplyPendingRestoreAsync` copies the *current* database to a dedicated `backups/pre-restore/coffer-{stamp}.db` set (same `BackupSnapshotWriter` copy technique, same 90-day rotation as pre-migration). So a restore of the *wrong* snapshot is not a dead end — the pre-restore copy is the undo. Correctness of financial data over everything: we never destroy the current state without first preserving it.
- **The swap copies the full WAL set, atomically, `.db` last.** Restoring reproduces the snapshot's on-disk set: copy the snapshot's `coffer.db` + `-wal` + `-shm` over the live paths, and **remove any live side-file the snapshot lacks** (a stale `-wal` belonging to the replaced DB would corrupt the restored one). Writer writes each into place; the `.db` moves last so an interruption never leaves a half-restored DB mistaken for good — the marker is only deleted after the `.db` is in place, so an interrupted restore simply re-runs on the next start.
- **Restore composes with the existing startup migration.** A snapshot may carry an older schema than the current build. We do **not** special-case that here: once the file is swapped and the app continues, the existing startup path (`HasPendingMigrations` → confirm window → `MigrationRunner` with its mandatory pre-migration backup) upgrades it exactly as it would any out-of-date DB (doc 08: "Migrate to current schema if needed"). Restore just puts the right bytes in place; migration is orthogonal and already built.
- **Snapshot enumeration is new; retention parsing is reused.** `BackupService` today exposes only aggregate status (`GetStatusAsync`) — its per-file `EnumerateDailyDates()` is private. Restore needs the *list* (date + size per snapshot), so we add a public listing method that enumerates `backups/coffer-*.db` and reuses the already-public pure `BackupRetention.ParseDailyDate`. `BackupSnapshotWriter` (internal to Infrastructure) is reused directly by `RestoreService` (also Infrastructure) for both the safety snapshot and the swap.
- **Direct `System.IO` behind `IVaultPaths`, staging behind an interface.** Same as Sprint 23: no `IFileSystem` wrapper exists; hard rule #4 is satisfied at the path level (`IVaultPaths`). `RestoreService` injects `IVaultPaths` + `ILogger` and does raw file I/O under the folder it hands them. Tests swap in `TestVaultPaths` (temp dir) + a real DB fixture. The marker (de)serialisation is a small pure piece, unit-tested at the boundary.

## Approach — headless service first, then UI + startup glue

- **24-A — restore service + snapshot listing + pending-restore apply (headless).** `IRestoreService` (list snapshots, stage a restore = write+validate the marker, query/clear a pending marker, apply a pending restore = safety snapshot + swap) + the marker model and its pure (de)serialiser + a public snapshot-listing method on `IBackupService`. New DI. Tests over `TestVaultPaths` + a real DB fixture, including the doc-08 round-trip (build a DB → snapshot → mutate → stage+apply → the mutation is undone → the pre-restore safety copy holds the mutated state). No pixels, no windows.
- **24-B — Settings action + restore picker + startup apply.** A "Restore from snapshot…" button in the Settings panel opening a restore dialog (list of dated snapshots, a clear "this replaces your current data; a safety backup is kept" confirmation, a "restart required" outcome); the `App.axaml.cs` startup hook that applies a pending restore **before** the DB opens; the panel's placeholder restore note replaced. Fully localized (keys in both `.resx`, parity green). VM tests over fakes.

## Definition of Done

- **24-A (automated):** `RestoreService` lists dated snapshots newest-first; `StageRestoreAsync` writes a valid marker for a real snapshot and rejects a missing one; `ApplyPendingRestoreAsync` swaps a snapshot into place **after** taking a `pre-restore` safety copy, undoes an intervening DB mutation, removes a stale live `-wal`, is a no-op with no marker, and re-runs cleanly after an interrupted apply; the marker (de)serialiser round-trips and tolerates a malformed file.
- **24-B (automated):** `RestoreViewModel` surfaces the snapshot list and stages the selected one (signalling restart), handles empty/cancel; `SettingsViewModel` opens the restore dialog and reports the staged outcome; resource-key parity holds.
- **Manual:** run the app, make a visible change (e.g. re-categorise a transaction), Settings → Backup & Recovery → "Restore from snapshot…", pick an earlier snapshot, confirm, restart; on restart the app opens the restored DB (the change is gone) and `%LocalAppData%\Coffer\backups\pre-restore\` holds a safety copy of the pre-restore state; logging back in works with the same master password.
- **Whole-sprint:** the owner can restore the encrypted database from a chosen daily snapshot through a guided, reversible, single-confirmation flow, with the current state preserved first — the local-snapshot half of doc-08's restore flow. Restore-from-BIP39-seed stays the documented next step (Sprint 25).

## Open questions

- **Restore timing: apply at next startup vs live in-app?** → proposed **stage-then-apply-at-startup** (write a marker in Settings, swap the closed DB before login on the next launch). Confirm vs a live restore (which would need connection-pool clear + factory teardown + in-process restart and still risks a locked file).
