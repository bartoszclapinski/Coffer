**Sprint 25 closed restore from a BIP39 seed** — the biggest remaining hole in the security model, and the completion of doc 09's decoupled DEK. Sprint 5 built every crypto primitive but wired only the password channel: `SetupService` received the mnemonic and never used it, and `DekFile` (v1) stored one password-wrapped copy — so a forgotten master password meant **total, permanent data loss**. This sprint makes `dek.encrypted` genuinely **dual-wrapped** (`DekFile` v2, backward-compatible with v1): a second, seed-wrapped copy of the DEK lets it unlock by *either* the Argon2 master key *or* the PBKDF2 recovery key. `SetupService` now writes v2 (new vaults get seed recovery for free); a new `ISeedRecoveryService` recovers a forgotten password (`RecoverWithSeedAsync`: seed → DEK → re-wrap under a new master password → log in), enables the channel on an existing vault (`EnableSeedRecoveryAsync`: wrap the in-holder DEK → v2), and reports state (`IsSeedRecoveryEnabledAsync`); rewrites go through an atomic `DekFile.WriteReplaceAsync` (temp → move) so a crash never destroys the only key file. A login-window **"Forgot password? Restore from seed"** takes the 12 words + a new password; a Settings **"Enable seed recovery"** mints a *fresh functional* seed (the v1 setup seed was shown but never wrapped anything, so it is cryptographically inert) via a generate → display (screen-capture-blocked) → verify flow, discarding the inert old one. Typed `SeedRecoveryUnavailableException` (v1 vault) + `InvalidRecoverySeedException` (wrong seed); the fixed derivation passphrase `"Coffer"` is frozen once a v2 vault exists; **no seed, password, or key is ever logged** (hard rule #6). Three PRs (25-A DEK dual-wrap + service, headless — #200; 25-B restore-from-seed UI — #202; 25-C enable-in-Settings — #204). **Deferred:** fresh-install/foreign-machine/archive restore-from-seed (needs archive import), a password-gated "Show recovery seed", a logged-in master-password change, and Argon2 param upgrade-on-login; the manual DoD click-through needs a running app + a real vault + the real seed. **Total data loss on a forgotten password is no longer possible once seed recovery is enabled.**

**Sprint 24 closed restore from a local snapshot** — the other half of doc-08's restore flow, and the guided counterpart to Sprint 23's backup safety net. Restoring a daily snapshot is the *same vault* (same DEK, same master password), so it is a pure file swap of already-encrypted bytes — no crypto, no password re-entry. A new `IRestoreService`/`RestoreService` (behind `IVaultPaths`, reusing Sprint-23's internal `BackupSnapshotWriter` + public `BackupRetention.ParseDailyDate`) lists the dated snapshots, and restore runs **two-phase**: from the Settings **"Restore from snapshot…"** dialog the owner only *stages* a choice (a `restore-pending.json` marker), and the actual swap happens at the **next startup, before the database is opened**, against a closed file (the Windows pending-file-rename pattern) — sidestepping SQLite's file-lock/WAL problem. The swap is **reversible**: `ApplyPendingRestoreAsync` copies the current DB to `backups/pre-restore/` (90-day rotation) before overwriting it, drops any stale live `-wal`/`-shm`, and clears the marker only after the `.db` is in place (an interrupted apply re-runs cleanly). A restored older-schema DB flows straight into the existing startup-migration path; a vanished staged snapshot surfaces a recovery window (discard-and-continue / quit) rather than a silent swallow or a bricked app. The `SettingsViewModel` opens the picker through a new `IRestoreDialogService` seam (mirroring `IFilePicker`) so `Coffer.Application` stays framework-free. **No schema, no AI, no network.** Two PRs (24-A `IRestoreService` + snapshot listing + pending-apply, headless — #192; 24-B restore picker + Settings action + startup apply + i18n — #194). **Deferred:** restore-from-BIP39-seed and archive/foreign-machine restore (Sprint 25 — needs a `dek.encrypted` v2 dual-wrap format carrying a seed-wrapped DEK, since v1 stores only a password-wrapped copy), the monthly auto-test restore (now unblocked by the listing method), Layer-2 Google Drive as a restore source, and a "Show recovery seed" action; the 24-B manual DoD click-through needs a running app + a real DB + a real snapshot.

**Sprint 23 closed local backup** — the encrypted database now has an automatic safety net. A shared `BackupSnapshotWriter` copies `coffer.db` + its `-wal`/`-shm` side-files; the Sprint-4 `PreMigrationBackup` was refactored onto it (behaviour unchanged) and its folder now rotates at 90 days. A new `BackupService` (behind `IVaultPaths`, no DB/DEK needed — it copies encrypted bytes) writes a dated `backups/coffer-{YYYY-MM-DD}.db` snapshot atomically (`.tmp`→`File.Move`, `.db` last), idempotent within a day, pruning daily files past 30 days via a pure `BackupRetention`; a delayed startup `Task.Run` fires it ~5 min after the window opens. An `ArchiveExporter` writes a portable `.zip` (encrypted DB + `dek.encrypted` + a manifest) to a path the owner picks through a new `IFilePicker` save method. A Settings **Backup & Recovery** panel shows the last snapshot + count with "Backup now" and "Export full archive". Deterministic, file I/O behind `IVaultPaths` (no `IFileSystem` in the codebase); **no schema, no AI, no network.** Two PRs (23-A service + retention + shared writer + exporter, headless — #186; 23-B Settings panel + picker save + startup scheduler + i18n — #188). **Deferred:** all restore flows (from snapshot / from BIP39 seed — the next sprint), Layer 2 Google Drive (needs the not-started Phase-3 sync spine), monthly auto-test restore, mobile, a "Show recovery seed" action.

**Sprint 22 closed the next-month expense forecast** — the app now *anticipates* next month per category, not only tracks the current one. A deterministic `ExpenseForecastEngine` in a new `Coffer.Core/Forecasting` combines each category's **fixed** part (active `Outflow` `RecurringFlow`s attributed by `MatchCategoryId` whose cadence lands in the target month, via a `FlowSchedule.OccursInMonth` helper lifted from the Sprint-16 projection engine) and **variable** part (trailing-3-month per-category debits, excluding transactions whose merchant matches an active flow so the recurring portion isn't double-counted, averaged monthly), then rounds the total up to the nearest 10 zł as a **suggested budget limit**. `IExpenseForecastQuery` assembles the inputs (anchored on the latest transaction's month, targeting the next); the uncategorised/unattributed spend is a single null bucket shown but not budgetable. A dedicated **Prognoza / Forecast** page shows each category's fixed/variable/total, suggested vs current limit, and a one-click **"Ustaw jako budżet"** that upserts through the Sprint-20 `SetBudgetAsync` (so a limit set there appears on the Budgets page). **The engine calculates; nothing here calls AI** — no migration, no new cost. Two PRs (22-A engine + query, headless — #180; 22-B page + set-as-budget + i18n — #182). **Deferred:** the optional `GetExpenseForecast` chat tool, seasonality/trend modelling, accrual-aware and multi-month forecasting, auto-applying suggestions, multi-currency; the manual DoD click-through needs a running app + real recurring history.

**Sprint 21 closed the Sprint-20 tail (the optional 20-C)** — budget state is no longer trapped on the Budgets page. An `OverBudgetDetector` (`IAnomalyDetector`) raises a dismissable alert on the **Alerty** page when a category's month-to-date spend crosses its `CategoryBudget` limit (`BudgetZone.Over`) — one per (category, month) via the signature `over-budget:{categoryId}:{yyyyMM}`, idempotent across re-scans and never resurrected once dismissed. To keep the detector pure, `AnomalyDetectionService` runs the existing `IBudgetTrackingQuery.GetOverviewAsync` once and hands the `BudgetOverview` to an extended `AnomalyDetectionContext`; the detector just filters the over-zone lines and emits Polish-templated candidates (fires on the crossing only, not the approaching warning). A read-only `GetBudgetStatusTool` (`IChatTool`, parameterless like `GetGoals`) lets the assistant report the current month's per-category limit/spent/remaining/projected/zone plus the unbudgeted lines, grounded in the same engine. **Headless** — the Alerty page renders alerts generically so no UI changed, `ChatService` auto-discovers the tool, and alert text stays Polish-templated (runtime localization still deferred under #128); no migration, no new AI cost. One PR (21-A — #176). **Deferred:** a `Warning`/approaching alert, alert retraction when spend later drops below the limit, per-account/per-month tool parameters, budget-specific AI commentary; the manual DoD click-through needs a running app + real data + an API key for the chat leg.
